# 代码架构设计

## 1. 现状判断

当前代码还不能满足下面三类需求的通用化实现：

1. 输入特定格式图片后自动完成场景构建。
2. 通过配置文件设置行人生成区域、生成方式和初始分布。
3. 通过配置文件设置行人路线、阶段切换和分流概率。

现有实现的特点是“可跑通，但高度硬编码”：

- 场景几何曾直接在 `scenes.py` 中用数组切片构造；该硬编码入口现已删除，场景主入口已切换为配置驱动。
- 行人初始分布直接在场景构造函数内写死，通过 `spawn_mask` 和 `cfg.rho_init` 生成 `initial_rho`。
- 路线和阶段切换直接在 case 构造函数内写死，多阶段路线由 `groups`、`TransitionRule` 和 `initial_group_density` 手工拼装。
- `runner.py` 只负责消费已经构造好的 `scene` 和 `case`，并不负责解释外部配置或图片。

这套实现适合论文原型验证，不适合后续做“多场景、多配置、多实验批量化”。

## 2. 当前代码如何工作

### 2.1 场景构建

当前场景构建完全来自程序化几何：

- 三通道场景：
  - 外边界、中央竖墙、三个开口、齿状障碍、出口位置，全部在代码里按网格比例计算。
  - 输出 `walkable`、`exit_mask`、`channel_masks`、`probe_x` 等数组。
- 游览场景：
  - 中央墙、上下开口、左侧回流出口也都在代码里按比例写死。

当前没有：

- 图片转网格掩码的解析器。
- CAD / SVG / GeoJSON 导入器。
- 外部场景配置文件。
- “同一求解器 + 多种场景输入源”的统一接口。

### 2.2 行人生成

当前行人生成只有“初始密度一次性赋值”这一种方式：

- 在场景函数里创建 `spawn_mask`。
- 将 `spawn_mask` 区域赋值为 `cfg.rho_init`。
- `runner.py` 在仿真开始时把 `scene.initial_rho` 或 `case.initial_group_density` 作为初始状态载入。

当前没有：

- 持续入流边界。
- 多个生成区并行配置。
- 按时间段变化的入流率。
- 不同群体不同初始分布的配置文件描述。

### 2.3 路线设置

当前路线设置分两类：

- 单阶段场景：
  - 用 `allowed_mask` 控制单向约束。
  - 用 `m11/m12/m22` 控制几何引导方向。
- 多阶段场景：
  - 用 `GroupModel.goal_mask` 指定每个群体当前目标区域。
  - 用 `TransitionRule` 指定阶段切换和固定概率分流。
  - 用 `initial_group_density` 指定初始群体。

当前没有：

- 从配置文件定义 route graph。
- 从配置文件定义 stage / route / decision zone / sink。
- 通用的“路径点序列 -> 群体目标场”编译器。
- 可复用的路线编辑格式。

## 3. 新架构目标

新的代码架构要解决的不是“再加几个 case”，而是把“场景输入、行人输入、路线输入”三件事从求解器里剥离出来，变成稳定的数据接口。

目标如下：

1. 同一仿真内核支持三种场景来源：
   - 程序化场景
   - 图片场景
   - 纯配置场景
2. 行人生成可通过配置文件描述：
   - 初始密度块
   - 多群体初始分布
   - 持续入流边界
   - 时间分段入流
3. 路线可通过配置文件描述：
   - 单阶段出口目标
   - 多阶段目标序列
   - 分流概率
   - 决策区和转移规则
4. 继续兼容现有 `runner.py` 和 `core.py` 的求解主链路，避免一次性推倒重写。

## 4. 目标架构总览

建议拆成 5 层：

1. `spec` 层
   - 负责定义外部输入的数据结构。
   - 例如 `SceneSpec`、`PopulationSpec`、`RouteSpec`、`ExperimentSpec`。
2. `loader` 层
   - 负责从 `toml/json/png` 读入并校验原始数据。
3. `compiler` 层
   - 负责把 `spec` 编译成当前求解器可消费的 `BaseScene` 和 `CaseModel`。
4. `simulation` 层
   - 沿用现有 `runner.py`、`core.py`、`metrics.py`。
5. `application` 层
   - 提供 CLI、批量实验入口、结果导出和后续优化入口。

推荐目录结构：

```text
codes/
  crowd_bellman/
    core.py
    runner.py
    metrics.py
    scenes.py
    spec/
      scene_spec.py
      population_spec.py
      route_spec.py
      experiment_spec.py
    loaders/
      scene_loader.py
      image_scene_loader.py
      config_loader.py
    compilers/
      scene_compiler.py
      population_compiler.py
      route_compiler.py
      policy_compiler.py
    app/
      simulate_from_config.py
      build_scene_from_image.py
      validate_config.py
  scenes/
    examples/
      three_channel.toml
      tour_scene.toml
      scene_legend.toml
```

## 5. 场景输入设计

### 5.1 支持的输入模式

场景输入建议支持三种模式：

1. `procedural`
   - 如需恢复程序化场景，应通过新的 `spec + compiler` 链路实现，而不是恢复旧的硬编码函数。
2. `image`
   - 输入一张标注图片和一份图例配置，自动栅格化为网格掩码。
3. `config`
   - 不依赖图片，纯配置文件直接定义矩形、多边形、折线障碍和区域。

### 5.2 图片格式约定

建议第一版支持 `PNG`，并采用“颜色语义映射 + 规则图层”的简单格式。

输入文件：

1. `scene.png`
   - 每个像素表示一个网格单元的类别。
2. `scene_legend.toml`
   - 定义颜色和语义的映射关系。

建议颜色语义：

- `#FFFFFF`：可行域
- `#000000`：障碍物
- `#FF0000`：出口
- `#00FF00`：生成区
- `#0000FF`：目标区 / 决策区候选
- `#FFFF00`：通道观测区

第一版不要直接从图片推导复杂路线语义，而是：

- 图片负责几何。
- 配置文件负责生成规则、路线规则、区域命名和参数。

这是必要拆分。只靠图片颜色无法稳定表达多阶段 route graph。

### 5.3 图片场景编译结果

图片输入编译后应输出：

- `walkable`
- `obstacle_mask`
- `exit_masks`
- `spawn_masks`
- `zone_masks`
- `probe_definitions`
- `named_regions`

其中 `named_regions` 是后续人口和路线配置的锚点，不能只返回匿名 mask。

## 6. 行人生成配置设计

### 6.1 目标

行人生成要从“单一 `rho_init`”升级为“可配置的人口注入系统”。

建议区分两类：

1. 初始人口
   - 仿真开始时一次性加载。
2. 持续入流
   - 仿真过程中按时间与边界规则持续注入。

### 6.2 推荐配置结构

建议将人口配置写入 `population.toml`，例如：

```toml
[[population.initial_groups]]
group_id = "visitors_main"
region = "spawn_left"
density = 2.2
route = "route_main_exit"

[[population.initial_groups]]
group_id = "tour_group"
region = "spawn_lower"
density = 1.8
route = "route_tour"

[[population.inflows]]
group_id = "late_arrivals"
boundary = "west_gate"
rate = 0.35
start_time = 20.0
end_time = 80.0
route = "route_main_exit"
```

### 6.3 编译规则

`population_compiler.py` 负责：

1. 将 `region` 或 `boundary` 名称解析为 mask。
2. 生成 `scene.initial_rho` 或 `case.initial_group_density`。
3. 生成运行时需要的 `inflow_rules`。

为了兼容现有内核，第一阶段可先实现：

- 多群体初始分布
- 多生成区初始分布

持续入流需要在 `runner.py` 中额外增加“每步注入”钩子，这属于第二阶段改造。

## 7. 路线配置设计

### 7.1 当前问题

现在路线是“直接写代码生成 `GroupModel` + `TransitionRule`”，这导致：

- 每新增一个场景都要改 Python 代码。
- 路线逻辑和几何逻辑混在一起。
- 无法做批量实验配置扫描。

### 7.2 推荐路线表达

路线配置建议采用“节点 + 转移”的图结构，而不是写死为 case 函数。

推荐结构：

```toml
[[routes]]
route_id = "route_main_exit"
mode = "single_stage"
goal_region = "exit_east"

[[routes]]
route_id = "route_tour"
mode = "multi_stage"
stages = ["stage_enter", "stage_tour", "stage_split", "stage_return"]

[[route_stages]]
stage_id = "stage_enter"
goal_region = "platform_upper"
sink_region = ""
allowed_directions = "all"

[[route_stages]]
stage_id = "stage_split"
goal_region = "lower_platform"
decision_region = "lower_platform"
kappa = 1.8
targets = [
  { stage = "route8", probability = 0.2 },
  { stage = "route9", probability = 0.3 },
  { stage = "route10", probability = 0.5 }
]

[[route_stages]]
stage_id = "route8"
goal_region = "exit_band_8"
next_stage = "stage_return"
```

### 7.3 路线编译器职责

`route_compiler.py` 负责把上面的配置编译成当前求解器对象：

1. 每个 `stage` 编译成一个 `GroupModel`。
2. `goal_region` / `sink_region` 解析成 mask。
3. `allowed_directions` 编译成 `allowed_mask`。
4. 引导方向规则编译成 `m11/m12/m22`。
5. `targets` 编译成 `TransitionRule`。
6. 根据 `population` 中绑定的 `route_id` 生成 `initial_group_density`。

### 7.4 单向约束和几何引导的配置化

路线配置还应支持把控制策略从代码参数变成配置参数，例如：

```toml
[control.geometry]
mode = "target_guidance"
alpha = 10.0
beta = 0.2

[control.direction]
mode = "one_way"
allowed = ["E"]
```

这样 `M(x)` 和 `U(x)` 的生成也能配置化，而不是只靠 `case2/case3/case4` 这种固定写法。

## 8. 配置文件体系

建议采用一个主配置文件组织全部输入，拆分子文件按需引用。

推荐形式：

```toml
[simulation]
nx = 96
ny = 72
dx = 0.5
steps = 2400
time_horizon = 160.0

[scene]
source = "image"
image = "scenes/example/scene.png"
legend = "scenes/example/scene_legend.toml"

[population]
file = "scenes/example/population.toml"

[routes]
file = "scenes/example/routes.toml"

[outputs]
output_root = "codes/results/custom_run"
```

第一版建议使用 `TOML`：

- Python 标准库可直接读取。
- 比 JSON 更适合人工编辑。
- 不强依赖额外第三方库。

## 9. 运行时接口设计

需要新增一个统一入口：

```text
python codes/simulate_from_config.py --config scenes/examples/run.toml
```

执行流程：

1. 读取 `run.toml`
2. 加载 `scene`
3. 加载 `population`
4. 加载 `routes`
5. 编译为 `BaseScene` + `CaseModel`
6. 调用现有 `simulate_case()`
7. 输出结果

这样可以保证：

- 现有求解器复用
- 外部输入规范化
- 后续参数搜索仍可复用同一套接口

## 10. 与现有代码的兼容方案

旧版程序化硬编码场景入口已删除。当前更稳妥的做法是：

1. 保留现有硬编码 case 作为回归测试样例。
2. 新增配置化入口。
3. 让原有硬编码 case 逐步迁移成“由内置 spec 生成”。

迁移后理想状态：

- `scenes.py`
  - 只保留少量内置 demo spec 或兼容函数。
- `spec/` + `loaders/` + `compilers/`
  - 负责全部输入解释。
- `runner.py`
  - 保持数值求解主逻辑。

## 11. 分阶段实施建议

### Phase 1

先打通配置文件，不碰图片输入：

1. 定义 `SceneSpec` / `PopulationSpec` / `RouteSpec`
2. 支持纯 `toml` 场景
3. 支持多生成区初始分布
4. 支持多阶段路线配置
5. 新增 `simulate_from_config.py`

### Phase 2

引入图片场景：

1. 支持 `png + legend.toml`
2. 生成 named regions
3. 将 population / routes 绑定到 named regions

### Phase 3

扩展运行时行为：

1. 持续入流
2. 更复杂的边界条件
3. 路线动态切换
4. 更丰富的控制策略参数化

## 12. 本次设计结论

结论很明确：

1. 当前代码不能直接实现“输入图片自动构建场景”。
2. 当前代码不能通过配置文件完成通用的人群生成配置。
3. 当前代码不能通过配置文件完成通用的路线配置。
4. 当前代码的求解器内核可以保留，但场景层、人口层、路线层必须重构为 `spec + loader + compiler` 架构。

后续实现时，应优先做：

1. `simulate_from_config.py`
2. `SceneSpec / PopulationSpec / RouteSpec`
3. `route_compiler.py`
4. `image_scene_loader.py`

否则你每加一个实验场景，都还会继续在 `scenes.py` 里写一个新函数，这条路很快会失控。
