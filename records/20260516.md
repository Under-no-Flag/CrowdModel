## 实施更新：重构安全指标并重跑 G2-v2

确认原实现仍使用硬阈值安全暴露：

```text
J2 = int 1_{rho > rho_safe} dxdt
```

在当前 G2-v2 正式结果中，硬阈值口径下 `J2_normalized` 约为 `0.0012-0.0022`，明显小于 `J1≈0.62` 和 `J5≈0.66-0.90`。该问题仍然存在：硬阈值二值计数只记录是否超阈值，不能区分超阈值幅度；即使改为归一化时空覆盖率，也会在等权目标中被压低。

已完成修改：

- `ObjectiveConfig` 增加 `j2_metric` 和 `j2_gamma`，当前默认 `j2_metric="soft"`、`j2_gamma=1.0`。
- `metrics.py` 增加 `safety_risk_density()`，soft 口径为 `((rho-rho_safe)_+/rho_safe)^gamma`。
- `record_step()` 使用 soft 风险密度累积 `J2`，并保留旧字段名 `j2_high_density_exposure` 作为兼容别名，同时新增 `j2_safety_risk` / `j2_safety_risk_normalized`。
- `G2StrategyCollector` 的热点和通道风险暴露同步改为同一 soft 口径。
- `g2_capacity_response_runner.py` 的生成配置固定写入 `j2_metric="soft"`、`j2_gamma=1.0`、`j2_scale=0.001`，并在 CSV / markdown 表中输出 `J2_eval`。
- `codes/scenes/examples/g2_multistage_directional/run_baseline.toml` 已同步为 soft J2 配置。

正式重跑命令：

```powershell
D:\Anaconda\envs\interpreter\python.exe codes\g2_capacity_response_runner.py --output-root codes/results/g2_capacity_response --steps 600 --time-horizon 35 --bellman-every 5 --save-every 100000 --density-contour-levels off
```

最终结果：

- `allow_g4_v2_optimization=true`。
- `J2_soft` 相对变化为 `0.726902`，高于硬阈值口径下的 `0.486878`，说明 soft 指标对容量策略差异更敏感。
- 原始归一化 `J2_soft` 范围约为 `0.000174-0.000466`，数值仍小；这是平均相对超密度的自然量级，不应直接等权加和。
- 经 `j2_scale=0.001` 标定后，`J2_eval` 范围约为 `0.174-0.466`，已经与 `J1≈0.62`、`J5≈0.66-0.90` 处于可共同驱动优化的量级。
- `J5` 相对变化为 `0.284912`，`J_B` 相对变化为 `0.489792`，拒绝通量相对变化为 `1.391538`。
- `max_cap_removed_relative=0.005567`，仍低于 `0.02` 阈值。

结论：用户提出的 soft J2 修改可行，但必须同时保留尺度标定。否则 `J2_soft` 虽然更连续、更能区分超阈值程度，原始归一化值仍会小于 `J1/J5`，固定权重优化中仍可能缺乏驱动力。
