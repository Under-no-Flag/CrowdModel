# 2026-05-17

## 实施：根据 `methodology/optimization v2.md` 实现新模型优化算法与实验

新增文件：

- `codes/crowd_bellman/g5_hcmbo.py`
- `codes/g5_runner.py`

实现内容：

- 新增 V2 控制向量 `z=(s,q)`，不覆盖旧 G4 的 `ControlVector(directions, eta)`。
- 固定 `eta0=(8,8,8,8)`，将连续优化变量从 `eta` 替换为内部入口容量 `q`。
- 根据方向状态约束 `q`：
  - `E` 只允许 `plus` gate；
  - `W` 只允许 `minus` gate；
  - `FREE` 同时允许 `plus/minus`，并用 simplex 映射控制双向总容量；
  - `CLOSED` 容量为 0。
- 实现 all-FREE no-cap 参考，用自然尝试进入流量估计 `qbar`。
- 实现分层优化流程：
  - 方向候选生成；
  - 低保真筛选；
  - 固定方向下的可行容量映射 `q=T_s(x)`；
  - 轻量 BO/LCB 候选生成；
  - DFO 坐标抛光；
  - 高保真复验。
- 实现 V2 标量目标：

```text
J = lambda1*J1_eval + lambda2*J2_eval + lambda5*J5_eval + lambdaB*J_B_norm + lambdaR*J_R_norm + penalty
```

默认权重为 `lambda=(1,1,1,1,0.1)`，`j2_scale=0.001`。

- `J1=0.596902`
- `J5=0.284410`
- `J2_eval=0.570091`
- `J_B_norm=0.082251`
- `J_R=1.224440`

结论：

- V2 优化算法与实验链路已经可运行。
- 当前默认权重与小预算下，有限容量控制没有优于 no-cap；它改善了效率和均衡，但显著牺牲安全、等待和平滑性。
- 论文中当前阶段不能写成“G5-v2 优化优于基线”，只能写成“实现了可行搜索，并揭示了 `q` 控制的目标冲突”。
- 下一步需要扩展预算，并做权重敏感性实验，例如提高 `lambda5`、降低/提高 `lambdaB`、测试 `lambdaR=0/0.1/1`。
